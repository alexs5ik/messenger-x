## 0. TL;DR для основателя

**Почему пользователь удалит Telegram и перейдёт к нам — за один абзац.**
Не из-за «ещё одного AI». А потому что Messenger X — единственное место, где (1) приватность доказуема постквантовой криптографией, которую конкуренты ещё не внедрили в групповых чатах, (2) AI работает *внутри* твоих зашифрованных данных через confidential compute, а не «читает всё в облаке», и (3) ты можешь **зарабатывать прямо в приложении** — продавать доступ к сообществам, контент, своих AI-агентов — с комиссией кратно ниже App Store. Telegram даёт (3) частично и не даёт (1)+(2); WhatsApp даёт приватность, но запрещает заработок; Discord даёт сообщества, но не приватность и не деньги. **Messenger X = пересечение всех трёх, которого нет ни у кого.**

**Три стратегических столпа (USP):**
1. **Provable Privacy** — PQ-крипто + MLS-группы + confidential AI (TEE), а не «доверьтесь нам».
2. **AI that respects the envelope** — AI работает по правилу «plaintext только в аттестованном анклаве или на устройстве».
3. **Creator OS** — встроенный заработок с комиссией 5-10% против 30% сторов.

**Главный технический вывод исследования:** E2E-шифрование и серверный AI находятся в *фундаментальном конфликте* (✅ Matthew Green, JHU; ✅ NYU/Cornell paper IACR 2024/2086). Это не баг нашего дизайна — это нерешённая проблема всей индустрии, и тот, кто решит её первым архитектурно (через Apple-PCC-подобный confidential compute), получает категориальное преимущество. **Это наш ров.**

# РАЗДЕЛ 1. PRODUCT VISION

**Видение.** Messenger X — операционная система общения и заработка: единое приложение, где человек общается, строит сообщества, продаёт контент и делегирует рутину AI-агентам, не жертвуя приватностью. Через 10 лет — это не «чат с AI», а **личный агентный слой**: пользователь пишет «купи билеты в Варшаву» — его агент ведёт переговоры с агентом авиакомпании (A2A) и завершает сделку.

**Три горизонта (Three Horizons модель):**
- **H1 (год 1-2): Лучший приватный мессенджер + AI в чате.** Захват ядра — переписка, звонки, AI-суммаризация, перевод. Метрика: retention D30.
- **H2 (год 2-4): Creator OS + сообщества + платежи.** Монетизация. Метрика: GMV и доля платящих создателей.
- **H3 (год 4-7): Agentic layer + A2A экономика.** Агенты совершают транзакции. Метрика: agent-completed tasks.

| Критерий | Оценка |
|---|---|
| **Плюсы** | Чёткий ответ на «зачем удалять Telegram»; defensible ров (privacy+AI); три источника выручки |
| **Риски** | Распыление (super-app вне Китая не взлетал — ✅ CNBC 2025); «privacy+AI» воспринимается как оксюморон и требует образования рынка |
| **Стоимость** | Vision сам по себе $0; цена — дисциплина не строить всё сразу |
| **Сроки** | Зафиксировать за 2-4 недели до первой строки кода |
| **Приоритет** | **P0 — критический.** Без него проект «обречён независимо от технологий» (формулировка заказчика верна) |
| **Альтернативы** | (а) Узкий privacy-мессенджер (Signal-клон) — нет монетизации; (б) AI-companion app — нет сетевого эффекта; (в) выбранный super-app путь — высокий риск/высокая награда |

**Принцип:** клиент-центричная криптография (plaintext рождается и умирает на устройстве) + stateless edge + горизонтально масштабируемое ядро + изолированный confidential-AI-слой.

└───────────────┬───────────────────────────┬────────────────┘

        │  EDGE GATEWAY  │          │  SFU MEDIA EDGE  │

   │            SERVICE MESH (Rust microservices)          │
   │  Auth │ Messaging │ Groups(MLS) │ Presence │ Media   │
   │  Payments │ Ads │ Communities │ Search │ Notif       │
   └──┬────────┬──────────┬──────────┬─────────┬──────────┘
      │        │          │          │         │
  ┌───▼──┐ ┌──▼───┐  ┌────▼───┐  ┌──▼──┐  ┌───▼────┐

# РАЗДЕЛ 6. DATABASE DESIGN

| Хранилище | Что хранит | Почему |
|---|---|---|
| **PostgreSQL** (шардированный, Citus/собств.) | Аккаунты, prekey-bundles, MLS-state, метаданные групп, подписки, кошельки | ACID, реляционная целостность для денег/идентичности |
| **Redis** | Presence, typing, rate-limits, сессии, кэш, очереди delivery-ack | Скорость, ephemeral |
| **Kafka** | Лог событий доставки, audit-trail, ad-events | Durable, replayable, fan-out |
| **ClickHouse** | Аналитика, метрики рекламы, агрегаты сообществ | Колоночная, миллиарды событий |
| **MinIO/S3** | Зашифрованные blob'ы (медиа, файлы) | Объектное хранилище, дёшево |
| **NATS** (не БД) | Internal RPC | Низкая латентность |

**Критический принцип:** **сервер хранит шифротекст, не plaintext.** Сообщения — это зашифрованные конверты (sender не доверяет серверу). ⚠️ Важно: исследование **опровергло** наивную модель «сервер вообще ничего не видит» (0-3) — метаданные (кто-кому-когда), cloud-backup, и push-routing неизбежно проходят через сервер. Мы это честно документируем и минимизируем (sealed sender, metadata-минимизация).

**Шардирование:** по `user_id` (hash) для Postgres; гео-шардирование для data-residency (GDPR).

| Критерий | Оценка |
|---|---|
| **Плюсы** | Каждое хранилище под свою задачу; шифротекст-only снижает риск утечки |
| **Риски** | Шардирование Postgres на 1B пользователей — сложно; cross-shard запросы; метаданные = поверхность атаки/регуляторного давления |
| **Стоимость** | $0.5-3M/год инфра (растёт с DAU) |
| **Сроки** | Схема ядра 2-3 мес; шардирование — год 2 |
| **Приоритет** | **P0** |
| **Альтернативы** | Cockroach/Spanner (проще шардинг, дороже); ScyllaDB вместо части Postgres |

# РАЗДЕЛ 7. SECURITY MODEL

**Это наш ров. Строим по эталону Signal и превосходим в группах + PQ.**

**Криптографический стек (всё верифицировано ✅):**
- **1:1 чаты:** X3DH/**PQXDH** установление ключа — гибрид **X25519 + CRYSTALS-Kyber/ML-KEM** (✅ Signal, 19 сен 2023: «атакующий должен взломать оба»). Далее — **Double Ratchet** → мигрируем на **post-quantum Triple Ratchet (SPQR)** с **ML-KEM-768** (✅ Signal, 2 окт 2025: forward secrecy + post-compromise security против квантового противника).
- **Группы/сообщества:** **MLS (RFC 9420)**, архитектура RFC 9750 — древовидное управление ключами O(log n), forward secrecy, post-compromise security, группы от 2 до десятков тысяч (✅; «сотни тысяч» — аспирационный потолок, не гарантия — закладываем реалистично десятки тысяч на криптогруппу, выше — через server-assisted каналы).
- **PQ-стандарты:** ✅ NIST финализировал **FIPS 203 (ML-KEM/Kyber)**, **FIPS 204 (ML-DSA)**, **FIPS 205 (SLH-DSA)** 13 авг 2024 — это settled-основа. Используем ML-KEM для KEM, ML-DSA для подписей.

**Принцип «Harvest now, decrypt later»:** PQ внедряем **с первого дня** — противник уже сейчас собирает шифротекст для будущей расшифровки квантовым компьютером. Гибрид (классика+PQ) защищает от обоих миров.

**Дополнительно:** sealed sender (скрытие метаданных отправителя), reproducible client builds, открытый код клиента, регулярный внешний аудит, key-transparency (как у WhatsApp/iMessage) для защиты от MITM-подмены ключей.

| Критерий | Оценка |
|---|---|
| **Плюсы** | Превосходит Telegram (нет E2E by default) и догоняет/обгоняет Signal в группах+PQ; математически доказуемые гарантии |
| **Риски** | Крипто-ошибки катастрофичны → нужен топ-крипто-найм + аудит; PQ увеличивает размер ключей (ML-KEM-768 EK 1184 байта) → трафик/батарея; **конфликт с AI** (раздел 8) |
| **Стоимость** | Крипто-команда (3-5 экспертов) + аудиты: $1-2M/год |
| **Сроки** | 1:1 PQXDH — 6 мес; MLS-группы — 9-12 мес; Triple Ratchet — год 2 |
| **Приоритет** | **P0 — это USP** |
| **Альтернативы** | Лицензировать libsignal (быстрее, безопаснее, чем своя крипта — **рекомендуется** не изобретать); MLS-реализации: OpenMLS (Rust) |

# РАЗДЕЛ 8. AI INFRASTRUCTURE

**Самый важный и самый сложный раздел. Здесь зарыт главный конфликт всего проекта.**

**✅ ВЕРИФИЦИРОВАННЫЙ ФУНДАМЕНТАЛЬНЫЙ КОНФЛИКТ:** E2E-шифрование и серверный AI, читающий контент, находятся в прямом противоречии. Источники:
- ✅ **Matthew Green (JHU), 17 янв 2025:** как только мощный general-purpose агент развёрнут на твоём телефоне, «остаётся лишь вопрос — кому разрешено с ним говорить»; и «значительная часть обработки будет вынесена на удалённые серверы», где «удалённая машина получит конфиденциальные данные в открытом виде».
- ✅ **NYU/Cornell (Knodel, Fabrega, Park, IACR 2024/2086, дек 2024):** интеграция AI «может подорвать конфиденциальность, которую обещает E2EE».
- ⚠️ Важно: тезис «E2E **абсолютно несовместим** с облачным AI» был **ОПРОВЕРГНУТ** (1-2). Точная формулировка — это *устранимое напряжение*, не несовместимость.

**Наше архитектурное решение — трёхуровневый AI (Tiered AI):**

```
Tier 1 — ON-DEVICE AI (по умолчанию, бесплатно)
  Малые модели на устройстве. Plaintext НЕ покидает телефон.
  Задачи: суммаризация чата, умные ответы, перевод, поиск по локальным данным.
  → 80% AI-фич, ноль конфликта с E2E.

Tier 2 — CONFIDENTIAL CLOUD AI (платно, для тяжёлых задач)
  Аттестованный TEE-анклав (модель Apple PCC ✅, июнь 2024:
  «данные недоступны никому, даже Apple», attestation + transparency log).
  Plaintext существует ТОЛЬКО внутри анклава, верифицируемо.
  Задачи: длинные документы, сложный анализ, RAG по большой истории.

Tier 3 — EXTERNAL FRONTIER AI (явное согласие, для агентов)
  Claude/GPT через API — ТОЛЬКО когда пользователь явно делегирует
  («купи билеты»). Данные покидают периметр осознанно.
```

**Правило envelope:** plaintext допустим только (а) на устройстве пользователя, (б) в аттестованном анклаве, (в) при явном осознанном согласии для внешнего агента. Никакого «тихого» чтения серверным AI.

**Экономика (⚠️ ОЦЕНКА — исследование НЕ подтвердило точные цифры, перепроверить):**
«AI в каждом чате» бесплатно для всех = разорительно при серверном инференсе. Поэтому Tier 1 (on-device) — бесплатный массовый слой, Tier 2/3 — платный (раздел 11). Грубая прикидка: серверный LLM-вызов ~$0.001-0.03 за запрос ⚠️ → при сотнях млн DAU безлимитный серверный AI = десятки млн $/мес. **On-device-first — не только privacy, но и юнит-экономика.**

| Критерий | Оценка |
|---|---|
| **Плюсы** | Решает конфликт privacy↔AI архитектурно (ров); on-device-first держит экономику; tiered = монетизация встроена |
| **Риски** | Confidential compute незрел/дорог; on-device-модели слабее; «verifiable enclave» сложно объяснить пользователю; зависимость от silicon-вендора (доверие смещается в кремний) |
| **Стоимость** | $3-8M/год (ML-команда + GPU/TEE-инфра); растёт с adoption |
| **Сроки** | Tier 1 — 6-9 мес; Tier 2 (TEE) — год 2-3 (R&D-heavy); Tier 3 — год 1 (просто API) |
| **Приоритет** | **P0 для Tier1+Tier3, P1 для Tier2** |
| **Альтернативы** | (а) Чистый on-device (max privacy, слабее AI); (б) honest «AI = не E2E, мы говорим прямо» (проще, но убивает USP); (в) гомоморфное шифрование (пока непрактично для LLM) |

# РАЗДЕЛ 9. COMMUNITY PLATFORM

**Аналог Telegram + Discord, но с приватностью и заработком.** Типы: публичные / частные / **платные** / корпоративные / DAO.

**Технически:**
- Малые/средние приватные группы → **MLS E2E** (✅ до десятков тысяч).
- Большие публичные каналы/трансляции → server-assisted (вещание, не E2E — как Telegram-каналы; контент и так публичен).
- Форумы/threads → как Discord forum channels.

**AI Communities (наша фича):** автомодерация, антиспам, авто-ответы новичкам, авто-FAQ — на on-device/анклавных моделях для приватных, серверных для публичных.

**Монетизация сообществ (раздел 11):** создатель продаёт подписки, курсы, файлы, консультации, доступ к своим AI-агентам.

| Критерий | Оценка |
|---|---|
| **Плюсы** | Захват и Telegram-, и Discord-юзкейсов; сетевой эффект; основа creator-экономики |
| **Риски** | E2E-модерация почти невозможна (нельзя сканировать то, что не видишь) → давление регуляторов (CSAM, DSA); масштаб MLS-групп ограничен |
| **Стоимость** | $2-4M (год 2-3) |
| **Сроки** | Базовые группы — год 1; платные сообщества — год 2; AI-модерация — год 2-3 |
| **Приоритет** | **P1** (ядро H2) |

# РАЗДЕЛ 10. ADVERTISING PLATFORM

**Не как у всех. Privacy-preserving реклама.** Главный вызов: реклама обычно требует данных, а у нас E2E.

**Решение — on-device таргетинг:**
- Профиль интересов строится **на устройстве** (из локального поведения), не на сервере.
- Сервер шлёт **пул** объявлений → устройство выбирает релевантное локально (модель Apple/Brave-style private ads).
- Пользователь **сам управляет** категориями интересов (контроль = доверие).
- Реклама только в **публичных** зонах (каналы, лента), **никогда** в приватных E2E-чатах.

**Аукцион/биллинг** — серверный (ClickHouse для событий), но без привязки к личности (агрегированные конверсии,差oldifferential privacy).

| Критерий | Оценка |
|---|---|
| **Плюсы** | Совместимо с privacy-USP; «реклама, которой ты управляешь» — дифференциатор от Telegram-спама |
| **Риски** | On-device таргетинг = слабее точность = ниже eCPM → меньше выручки на показ; противоречие месседжа «приватный мессенджер С рекламой» |
| **Стоимость** | $2-5M (год 3+) |
| **Сроки** | Год 3 (после набора аудитории — реклама без масштаба бессмысленна) |
| **Приоритет** | **P3** (поздно, не на старте) |
| **Альтернативы** | Вообще без рекламы (как Signal) — чище USP, но теряем источник выручки; контекстная реклама без таргетинга |

# РАЗДЕЛ 12. SCALABILITY PLAN

**Цель: 1B пользователей. Доказано, что это достижимо малой командой при правильном стеке** (✅ Discord: 5 инж/12M concurrent; WhatsApp ~50 инж/900M ⚠️).

**Принципы:**
- **Stateless edge** (Gateway) — горизонтальное масштабирование за L4-балансировщиком.
- **Sharding по user_id** для Postgres; гео-шарды для residency.
- **Rust на горячих путях** (✅ доказанная эффективность — больше юзеров на ядро).
- **Cell-based architecture** (изолированные ячейки по N млн пользователей) — отказ одной ячейки не роняет всех.
- **Multi-region active-active**, выбор ближайшего edge по latency.
- **SFU cascade** для медиа (раздел 14).

**Этапы:** 0→1M (один регион, модульный монолит) → 1M→50M (микросервисы, шардинг) → 50M→500M (multi-region cells) → 500M→1B (глобальная mesh, edge-везде).

| Критерий | Оценка |
|---|---|
| **Плюсы** | Прецеденты доказывают достижимость; cell-архитектура = blast-radius контроль |
| **Риски** | Шардинг-миграции болезненны; глобальная консистентность; стоимость растёт нелинейно |
| **Стоимость** | Инфра: от $0.5M/год (1M) до $50-150M/год (1B) ⚠️ |
| **Сроки** | Поэтапно 5 лет |
| **Приоритет** | **P1** (закладывать принципы с P0, реализовывать по нагрузке) |
| **Альтернативы** | Managed (AWS-everything) — дороже, быстрее; bare-metal (Hetzner) — дешевле, сложнее |

# РАЗДЕЛ 13. CLOUD ARCHITECTURE

**Гибрид для экономики на масштабе.**

| Слой | Провайдер | Почему |
|---|---|---|
| **Edge/CDN/DDoS** | Cloudflare | Глобальный edge, защита, QUIC-нативность |
| **Compute ядро (масштаб)** | Hetzner / bare-metal | ✅ драматически дешевле AWS на стабильной нагрузке (мессенджер — предсказуемая нагрузка) |

| Критерий | Оценка |
|---|---|
| **Плюсы** | Минимизация cloud-косты (критично для freemium); no vendor lock-in |
| **Риски** | Гибрид = операционная сложность; bare-metal требует своей SRE-экспертизы; data-residency |
| **Стоимость** | Экономия 40-70% против all-AWS на масштабе ⚠️ |
| **Сроки** | Cloudflare+managed на старте; миграция ядра на bare-metal — год 2 при росте косты |
| **Приоритет** | **P1** |
| **Альтернативы** | All-AWS (просто, дорого); all-bare-metal (дёшево, рискованно для стартапа) |

# РАЗДЕЛ 15. MOBILE ARCHITECTURE (iOS + Android)

| Критерий | Оценка |

| **Риски** | Rust↔native FFI-сложность; размер бандла (PQ + on-device модели); фрагментация Android; review-политики сторов (крипто/платежи) |
| **Стоимость** | Mobile-команда 8-15 чел: $2-4M/год |

| **Приоритет** | **P0** (мобайл — основной рынок мессенджеров) |
| **Альтернативы** | Flutter/RN (быстрее, но хуже для крипто/perf/on-device AI — не рекомендуется для ядра); KMP вместо Rust-core |

# РАЗДЕЛ 16. WEB ARCHITECTURE

- **SPA на TypeScript** (React/Svelte) + **WASM-сборка Rust-крипто-ядра** — та же аудируемая крипта в браузере.
- **Транспорт:** WebTransport (QUIC) где поддерживается, WebSocket fallback.
- **Хранилище ключей:** IndexedDB + WebCrypto; честное предупреждение — браузер слабее для key-security (нет TEE), поэтому web = «удобство», мобайл = «максимальная безопасность».
- **On-device AI в вебе:** WebGPU для малых моделей (ограниченно).
- PWA для desktop-web установки.

| Критерий | Оценка |
|---|---|
| **Плюсы** | Zero-install onboarding; WASM = единая крипта; широкий охват |
| **Риски** | Браузер — слабейшее звено безопасности (XSS→ключи); reproducibility веб-крипты сложна (можно подменить JS) → нужен code-signing/SubResource Integrity |
| **Стоимость** | Web-команда 4-8 чел: $1-2M/год |
| **Сроки** | После мобайла — год 1-2 |
| **Приоритет** | **P1** |
| **Альтернативы** | Только Electron-desktop без чистого веба (безопаснее, меньше охват) |

| Маркетинг/рост | $2-10M | $30-60M | $100M+ |
| **Итого/год** | **~$10-25M** | **~$80-150M** | **~$265-460M** |

**Совокупно за 5 лет до 100M+ MAU: порядка $400M-1B** (сопоставимо с тем, что Telegram потратил годами и привлекал через долговые бонды/ICO).

**Выручка (потоки, год 3+):** Premium-подписки (ARPU paying ~$60/год × 3-5% конверсия), Creator OS take-rate (5-10% от GMV), Business API, реклама (год 3+), payments take-rate. **При 100M MAU и 4% платящих × $5/мес ≈ $240M/год** ⚠️ только с Premium, плюс Creator/Ads сверху.

**Финансирование:** глубокие раунды VC (privacy+AI — горячая тема), либо модель Дурова (собственные средства + бонды), чтобы не зависеть от рекламной монетизации на ранней стадии (приватность как ценность требует независимости).

| Критерий | Оценка |
|---|---|
| **Плюсы** | Множественные потоки; freemium-воронка; высокий ARPU потенциал у Creator/Business |
| **Риски** | Капиталоёмкость огромна; AI/инфра-косты могут опередить выручку; ⚠️ все цифры требуют валидации; super-app монетизация вне Китая буксовала |
| **Приоритет** | **P0** (без модели нет fundraising) |
| **Альтернативы** | Lean privacy-only (дешевле, медленнее); bootstrap через B2B-крипто-инфру |

# РАЗДЕЛ 20. ПЛАН ДОСТИЖЕНИЯ 100 МЛН ПОЛЬЗОВАТЕЛЕЙ

**Рынок атаки: весь мир, но через приоритетные регионы** (нельзя «весь мир сразу» — нужны точки опоры).

**Last-mover advantage:** не повторять путь WeChat (провал на Западе ✅ CNBC). Входить там, где боль приватности + желание заработка максимальны.

**Приоритизация регионов:**
1. **Privacy-чувствительные + tech-ранние** (EU, особенно после DMA — обход 30% сторов легален; Германия/Нидерланды): privacy-USP резонирует, регуляторика на нашей стороне против App Store.
2. **Creator-экономики + обход дорогих платёжек** (Юго-Восточная Азия, LatAm, Индия): Creator OS + дешёвые платежи = killer.
3. **Telegram-heavy миграция** (СНГ, Ближний Восток, крипто-сообщества): прямое позиционирование «Telegram, но реально приватный + PQ + заработок».

**Механика роста (loops):**
- **Creator-driven loop:** низкая комиссия (5-10% vs 30%) → создатели мигрируют → приводят свои аудитории → аудитории остаются ради контента. *Это главный двигатель* (как Discord рос через игровые сообщества).
- **Privacy-events loop:** каждый скандал утечки/бэкдора конкурента = окно миграции (Signal так растёт скачками).
- **AI-utility loop:** «суммируй чат / переведи / найди задачу» — daily-habit фичи, повышающие retention.
- **Referral + group-import:** импорт групп из Telegram/WhatsApp снижает switching cost.

**Последовательность к 100M:**
0→1M (год 1): privacy+crypto early adopters, HN/Reddit/крипто-комьюнити, PR вокруг PQ-крипты «первый PQ super-messenger».
1M→10M (год 2): Creator OS закрытая бета, привлечение якорных создателей с эксклюзивом.
10M→100M (год 3): региональная экспансия (EU→SEA→LatAm), реферальные циклы, открытие платежей.

| Критерий | Оценка |
|---|---|
| **Плюсы** | Региональная фокусировка реалистичнее «весь мир»; creator-loop = самый сильный из доказанных каналов; privacy-события = бесплатный рост |
| **Риски** | CAC высок против бесплатных гигантов; network-effect конкурентов (друзья в WhatsApp) = главный барьер миграции; ⚠️ цифры роста спекулятивны |
| **Стоимость** | $50-150M маркетинг к 100M ⚠️ |
| **Сроки** | 100M к концу года 3 — агрессивно, но прецеденты (TikTok/Telegram) есть |
| **Приоритет** | **P0** |
| **Альтернативы** | Один регион глубоко (ниже риск, медленнее); B2B-first (бизнес-мессенджер) как клин |
